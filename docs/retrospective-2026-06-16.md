## 오늘의 교훈 (Lessons)

1. **데이터 모델이 의사결정 기능의 키스톤이다.** 모든 값이 문자열일 때는 "어느 게 더 좋은지"를
   시스템이 알 수 없었다. `Spec`에 `kind/unit/betterDirection/weight`를 추가하자 승자 하이라이트·
   정규화·가중 점수·레이더가 **하나의 파서(`utils/specValue.ts`)**로 일관되게 따라왔다. 기능 추가
   전에 모델을 먼저 정리하는 게 정답이었다.

2. **이미지는 핫링크 대신 저장소 재호스팅.** 제조사 이미지를 직접 링크하면 끊기거나 차단된다.
   `public/products/<slug>.(svg|jpg)`로 두고 경로로 참조하니 안정적이고, 실제 사진은 같은 슬러그로
   덮어쓰기만 하면 된다. 슬러그 규칙은 데이터(`mockData`)와 생성기(`scripts`)가 **동일하게** 공유해야
   한다(불일치 시 이미지가 안 뜸).

3. **수집(브라우저)과 영속화(파일시스템)는 분리한다.** 크롬 확장은 사실·이미지 URL 추출에 강하고,
   파일 저장·재호스팅은 Claude Code가 한다. 확장 출력 스키마를 앱의 import 스키마와 정확히 맞춰두면
   왕복이 매끄럽다 → 그래서 프롬프트에 **정확한 스펙 필드명**을 박아두는 게 중요했다.

4. **빈 값은 지어내지 말고 비운다.** 1차 수집에서 일부 contrast/lightSource가 비어 왔는데, 추측으로
   채우는 대신 기존 시드 값을 유지하거나 `-`로 표시했다. "사실만" 규칙이 신뢰의 핵심.

5. **localStorage 캐시 함정.** 시드 데이터를 바꿔도 이미 방문한 브라우저는 옛 데이터를 들고 있다.
   "샘플 데이터" 버튼으로 강제 갱신 경로를 만들어 두길 잘했다.

6. **번들은 라우트가 아니라 무게로 쪼갠다.** recharts가 단독 ~360kB였는데, 차트 모달을 `React.lazy`로
   지연 로드하니 초기 번들이 절반 이하로 줄었다. "열 때만 필요한 무거운 의존성"이 1순위 분할 대상.

7. **검증 루프를 짧게.** 매 변경마다 `tsc --noEmit` + `npm run build`로 즉시 확인하고 단계별로
   커밋·푸시(Netlify 자동 배포)했다. 작은 단위 배포가 디버깅을 쉽게 했다.

8. **푸시 전 rebase 습관.** 원격(README 등)이 앞서 있어 push가 reject되곤 했다. `git fetch` →
   `git rebase origin/main` → push로 충돌 없이 정리. main 직접 배포 워크플로에서 특히 중요.
